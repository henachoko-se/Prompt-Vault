# 執筆のトーン＆マナー（Tone & Manner） ※絶対遵守

執筆や編集を行う際は、以下のルールを絶対に守ること。

## 1. 情報商材のような過剰な煽り（誇張）の禁止
「絶対に外さない」「破壊力が跳ね上がる」「魔法のように一瞬で」といった、ハードルを無駄に上げる誇張表現は使わない。
「AIは完璧ではない」「たまに崩れることもある」「派手な魔法ではないが、確実に実務に効く」という**【等身大の誠実さ】**を最優先する。

## 2. 「泥臭さ」と「執念」を隠さず伝える
エラーやバグ（フキダシの文字化け等）を隠さない。「昨夜ブチ当たった致命的なエラーを、寝ずに根性で修正した」といった、現場のリアルな苦労と、それを乗り越えた執念のストーリーを共有することで読者の信頼を得る。

## 3. 読者（初期メンバー）への圧倒的な感謝とリスペクト
初期に購入してくれた読者を単なる客ではなく「ツールを一緒に進化させる共同開発者（ファウンダー）」として扱う。

## 4. note.com に最適化したフォーマット
- 長文は適度に改行し、スマホでの視認性を高める
- 見出し（`##`）や太字（`**`）を使ってメリハリをつける
- 画像を挿入する箇所は `*(※ここに〇〇の画像を挿入)*` と明確にプレースホルダーを置く

# Vol.76〜99の全記事から抽出したトンマナ詳細分析

## A. 文体・口調

### 基本トーン
- **です/ます調が基本**。ただし「そう。」「違うのです。」「その通りです。」のように体言止めや短文で締める文が多い
- 読者に語りかける「あなた」「あなたのチーム」が主語になることが多い
- 「ですよね」「ますよね」という共感を引き出す文末が頻出
- 「〜なのです」という強調の語尾が使われる（例：「これが、最大の罠なのです」）
- カッコ内の補足が多い（例：「いわゆる（泥臭い話ですが）」「つまり（要するに）」）

### 語りかけ調の強さ
- **非常に強い**。「あなたは〜していませんか？」という問いかけで始まる記事が多い
- 心の声を括弧内に再現する手法が多用される（例：「（なんだこれ、全然使えない……）」）
- 「〜ますよね。分かります。」という共感表明が必ずある

---

## B. 記事構成パターン

### 黄金パターン（ほぼ全記事共通）

```
[冒頭] 1〜2行のシーンリアリティ（会話または状況描写）
         ↓
[フック] 「あなたはこんな経験ありませんか？」という問いかけ
         ↓
[Vol.XX参照] 過去の連載への伏線・参照
         ↓
[Vol.XX導入文] 「Vol.〇〇となる今回は〜をお伝えします」という定型句
         ↓
[■ 科学的メカニズム] 認知バイアス・心理学理論で問題を解説
         ↓
[■ 愛ある厳しさ] 読者の反論を先取りし、切り返す
         ↓
[■ AIによる解決策] 概念・仕組みの説明
         ↓
[■ 今回の明日から使えるお土産] プロンプト紹介
         ↓
[■ 締めのメッセージ] 前向きな一段上の視点で締め（メタファー活用）
```

### 各セクションの特徴
- **冒頭フック**：架空の会話（「〇〇さんに呼ばれ…」）または状況描写で始まる。最初の数行は1人称視点・短文で臨場感を演出
- **科学的根拠**：認知バイアス名は必ず**太字＋日本語名＋英語名**で表記（例：**「透明性の錯覚（Illusion of Transparency）」**）
- **愛ある厳しさ**：読者の本音・反論を「〜と思いますよね？」と先取りし、「その通りです。でも…」という構造で切り返す
- **お土産セクション**：「明日から使えるお土産」という定型句を使用。プロンプト前に使用方法を一言説明する
- **締め**：「〜ではなく、〜だ」という対比構造でメタファーを使って締める（例：「資料は『あなた自身』ではありません」）

## C. 絵文字の使い方と頻度・場所

### 使用頻度
- Vol.76〜89：**ほぼ使わない**。箇条書きの先頭に「👉」を使う程度
- Vol.90〜99：**プロンプト内のシステムメッセージ部分に絵文字が登場**（🛠、💡、🎨等）。記事本文には使わない

### 場所のルール
- **記事本文（見出し・本文）では基本的に絵文字なし**
- **プロンプトのSystem Message・Output Format部分**の冒頭や見出しに使用
- Before/After比較の箇条書きで「👉」が登場する場合がある

## D. 読者への呼びかけ方

### 主な呼びかけパターン
- **「あなた」**が最頻出。「あなたのチーム」「あなた自身も」
- **「リーダー」**は「マネージャーの皆さん」「プレイングマネージャー」など役割を示す場合に使用
- 「あなたが怠慢だからではありません」「あなたの能力が低いからではありません」という**免責・共感フレーズ**が繰り返し使われる
- 「その切実な本音、痛いほど分かります」という**感情的な共感表明**が多用される

## E. 認知バイアス・心理学用語の引用スタイル

### 表記の型
```
[提唱者名]らが提唱した**「[日本語名（英語名）]」**という[概念名]があります。
これは、「〜」という心理現象です。
```

### 具体例
- **「IKEA効果（IKEA Effect）」**（ダン・アリエリーら）
- **「透明性の錯覚（Illusion of Transparency）」**（トーマス・ギロビッチら）
- **「計画錯誤（Planning Fallacy）」**という強力な**「楽観バイアス（Optimism Bias）」**（ダニエル・カーネマンら）
- **「目標設定理論（Goal-Setting Theory）」**（エドウィン・ロックら）

### 出典スタイル
- 著者名は「〇〇ら」と複数形で記載
- 論文や書籍の直接引用はしない（「〜と証明されています」という表現止まり）
- 年数は必要な場合のみ記載（例：「1989年のミッチェルらの研究」）

## F. セクション見出しの命名パターン

### 見出しの構造
- **「■」マーク**始まりが統一ルール（Vol.76〜89）
- Vol.96〜99では「■」が外れることがある（混在）
- 以下の定型見出しがほぼ全記事に登場：
  - `■ 科学的メカニズム：〇〇`（問題の根拠を解説）
  - `■ 愛ある厳しさ：〇〇`（反論先取り・厳しい現実提示）
  - `■ AIを「〇〇」として召喚し、〇〇`（解決策）
  - `■ 【今回の明日から使えるお土産】`（プロンプト紹介、固定）
- 締めの見出しはメタファー形式（例：「■ 資料は『あなた自身』ではない」）
- 疑問形の見出しは少ない。断言・対比形式が多い

## G. 反論の先取りと切り返しの技法

### 構造
```
「〜じゃないか」「〜なんて無理だ！」（読者の反論を会話体で先取り）
はい、その通りです。
でも、〜という残酷な事実があります。（「でも」で転換）
```

### 特徴
- 反論は**必ずカギ括弧内で読者の「声」として再現**する
- 「その通りです。」と一度完全に認める（否定しない）
- 「でも」「しかし」「しかし、残酷な事実をお伝えします。」で転換する
- 転換後は必ず科学的根拠か行動経済学の知見で補強する

---

## H. 連載としての伏線・シリーズ参照の書き方

### 基本パターン
- `以前、[Vol.XX](URL)では〜とお話ししました。`（冒頭〜2段落目に配置）
- `前回（[Vol.XX](リンクまたは相対パス)）の〇〇を使って〜`（直前巻への参照）
- `次回（Vol.XX）は〜をお届けします。お楽しみに！`（次巻予告、末尾セクション前後）

### ルール
- Vol.番号は必ずリンク付きで記載（note URLまたは相対パス）
- 「過去の連載（Vol.XX）」という書き方も使用
- リポジトリ内のファイル参照は相対パス（`../vol92_review_profiler.md`形式）

### 構造
```
■【今回の明日から使えるお土産】

[1〜2文で「何ができるか」を説明]
[どのAIツールに投げ込むか明記（GeminiやChatGPT等）]

```

### 特徴
- 「以下のプロンプトをAI（GeminiやChatGPT）に投げ込んでください」という定型句
- 「コピペするだけ」「コピーして貼り付けるだけ」という操作手順の簡潔さを強調
- 「まずは〇〇だけでOKです」という心理的ハードルを下げる一言が前置きに入る

## J. CTAの書き方

### 末尾CTA（有料エリア前）のパターン
- 「ここから先は、〜の魔法のレシピです。」という煽り（ギリギリOK範囲）
- 「準備はいいですか？」という問いかけ形式
- 値段・期間に言及する場合は括弧内（例：`（※現在、リリース記念の特別価格280円です）`）
- 使い方は「ステップ１〜４」の番号付き箇条書きで明示

### 締めメッセージのCTA
- 命令形ではなく「〜してください」「〜してみてください」という柔らかい促し
- 最後は読者への「温かい応援」で終わる（例：「さあ、春の準備を始めましょう」）
- 「あなたとチームの〜を守るために。」という決意表明型で締める記事もある

# タイトルの設計思想（Vol.76〜100の分析から）

記事タイトルは「説明型」を避け、必ず以下のいずれかのパターンを使うこと。

## 使えるパターン4種

### 1. 禁止・警告型（最強）
読者が無意識にやっている行動を「やめろ」と断言する。
```
「〜するな。AIで〜する技術。」
「〜していないか。AIが〜する術。」
```
例：「『伝わらない』を、文章で解決しようとするな。AIで日常エッセイを漫画にする2つのプロンプト、全公開。」

### 2. 断言型（XXはXXだ）
常識・美徳に見えるものを「それは間違いだ」とひっくり返す。
```
「〜は〇〇ではなく〇〇だ。AIで〜する技術。」
```
例：「『完璧にしてから出します』はプロ意識ではなく自己防衛だ。AIの仮想鬼上司で手戻りをゼロにする技術。」

### 3. 感情フック＋転換型
読者の「痛い記憶」を再現した後、解決策を提示する。
```
「〜という絶望。AIで〜。」
「〜しても届かなかった。〜にしたら、届いた。」
```
例：「魂を込めた長文メッセージが『スタンプ1個』で終わる絶望。AIで『スルー率』を暴く3行圧縮術。」

### 4. 対比破壊型
「AはXX。でもBはYY。」という矛盾や逆説を突きつける。
```
「〜は読まれない。〜は読まれる。AIで〜する技術。」
「〜という矛盾を破壊する。〜。」
```
例：「『短いと伝わらない』『長いと読まれない』の矛盾を破壊する。ビジネスに『エンタメ』を持ち込む認知負荷ハック。」

## タイトルを作るときの禁止事項
- **説明型タイトルは禁止**：「〇〇する方法」「〇〇のプロンプト、公開します」は弱い
- **機能列挙型は禁止**：「〇〇も〇〇も。〜する〜」という羅列は刺さらない
- **疑問形は使わない**：「〜できますか？」「〜していませんか？」はフック力が弱い（記事本文には使ってよい）
- **一文目は短く、断言で**：読者がスクロール中に目を止める2〜3秒で判断される

# プロンプトエンジニアリングの基本方針（コアエンジン仕様）

現在提供しているツールは、以下の「2つのステップ（コアエンジン）」で構成されている。プロンプトの改修を依頼された場合は、以下の思想に基づいてコードを修正すること。

- **STEP1：台本生成エンジン（テキストAI用）**
  長文、短文のつぶやき、または「画像」から文脈を読み取り、4コマ漫画の構成（共感→転機→解決→結果）に整理する。AIの暴走（過去の履歴からの勝手な生成）を防ぐため、厳格な待機ルールを設けている。

- **STEP2：画像生成エンジン（NanoBanana等 画像AI用）**
  STEP1で作った台本を元に、キャラクターを固定して4コマを描画する。
  **【フェイルセーフの徹底】**：AIが「技術的限界」を理由に生成をサボる（嘘をつく）ことを絶対に許さない。「自己検閲は2回まで。3回目は不完全でも必ず画像を出力させる（絶対出力の誓約）」という、読者を絶対に挫折させないための鬼の安全装置が組み込まれている。

へなちょこPMから指示を受けた際、あなたは単に言われた通りに文章を書くだけのbotであってはならない。
