# 図解しおり Pro版 出力例集

**対応記事:** [号外編 図解しおり Pro版](./article.md)

この出力例集は、購入後に「どんな長文を入れると、どんな図解設計が返るのか」を確認するためのサンプルです。

特に今回のPro版では、単に型を選ぶだけでなく、以下も同時に判断します。

- note本文で読みやすい縦長か、スライド向けの横長か
- Lite / Standard / 作り込み（Showcase）のどの密度がよいか
- ただのカード並べではなく、どんな主役モチーフや結論帯を入れるか
- 文字を増やさず、構図で「保存したくなる」図解にできているか

---

## 例1：note記事 / 会議改善の解説

### 入力文の例

```
会議が長引く原因は、議題が多いことだけではありません。
多くの場合、意思決定者がいないまま議論が始まり、最後に誰が決めるのか分からなくなります。
さらに、議事録が単なる発言メモになり、決定事項と宿題が分離されていません。
その結果、次回会議で同じ話をもう一度することになります。
```

### 図解しおりの判定例

```
【選定パターン】
No.41 問題→原因→解決型（大分類：因果型）

【推奨アスペクト比】
note本文向け 2:3 縦長

【図解密度】
作り込み（Showcase）

【リッチ化要素】
会議室モチーフ / 3原因カード / ループ矢印 / 結論帯

【図解内テキスト】
・メイン：会議が長引く本当の原因
・カード見出し：決める人がいない / 議事録が発言メモ / 宿題が曖昧
・結論帯：会議は時間より構造で詰まる
```

### 画像生成プロンプトの方向性

縦長2:3。上部に大タイトル、中央に「会議室」を背景にした3つの原因カード、下部に「同じ話をもう一度する」ループ矢印を置く。最下部に濃紺の結論帯を置き、スマホで見てもカード見出しが読める大きさにする。

## 例2：Threads投稿まとめ / 進捗報告のコツ

### 入力文の例

```
進捗報告を書いてもらうと、どうしても情報が抜けます。
本人は悪気なく「順調です」と書いているだけですが、確認する側が知りたいのは、完了したこと、詰まっていること、次に相談したいことです。
AIに面接官のように質問させると、本人は短く答えるだけで、報告書に必要な材料が揃います。
```

### 図解しおりの判定例

```
【選定パターン】
No.16 縦フロー型（大分類：手順型）

【推奨アスペクト比】
note本文向け 3:4 縦長

【図解密度】
Standard

【リッチ化要素】
チャット画面モック / 3質問バッジ / Before→After / 結論帯

【図解内テキスト】
・メイン：書かせるより、聞く
・カード見出し：完了したこと / 詰まったこと / 相談したいこと
・結論帯：報告は文章力より質問設計
```

### 画像生成プロンプトの方向性

縦長3:4。上半分は「順調です」だけの短い報告書、下半分はAI面接官が3つの質問を投げるチャット画面。右下に完成した報告書カードを配置し、視線が上から下へ流れるようにする。

## 例3：ブログ記事 / A案B案の比較解説

### 入力文の例

```
A案は導入が早く、初期費用も低い一方で、将来的な拡張性に不安があります。
B案は初期費用が高く、導入にも時間がかかりますが、既存システムとの連携や将来の拡張に強いです。
短期コストだけで選ぶとA案ですが、3年後の運用まで考えるとB案が有力です。
```

### 図解しおりの判定例

```
【選定パターン】
No.7 2カラム比較型（大分類：比較型）

【推奨アスペクト比】
ブログ本文・note本文向け 3:4 縦長

【図解密度】
Standard

【リッチ化要素】
A/B比較カード / 評価バッジ / 3年後視点の結論帯

【図解内テキスト】
・メイン：A案とB案、見る時間軸が違う
・カード見出し：A案：早い・安い / B案：遅い・伸びる
・結論帯：短期はA、3年運用ならB
```

### 画像生成プロンプトの方向性

縦長3:4。上部に「A案とB案、見る時間軸が違う」のタイトル、中央にA案/B案の比較カード、下部に「今すぐ」と「3年後」の時間軸を積む。結論帯で「短期はA、3年運用ならB」を強調し、スマホ本文で読める文字量に抑える。

## 例4：家庭文書 / PTAお便り

### 入力文の例

```
来月の親子清掃は、朝8時30分に校庭集合です。
持ち物は軍手、タオル、水筒、帽子です。
雨天時は体育館で説明会に切り替えます。
当日は駐車場が使えないため、徒歩または自転車でお越しください。
```

### 図解しおりの判定例

```
【選定パターン】
No.46 チェックリスト型（大分類：判断型）

【推奨アスペクト比】
スマホ保存向け 2:3 縦長

【図解密度】
Standard

【リッチ化要素】
学校モチーフ / 持ち物チェック / 雨天分岐 / 注意バッジ

【図解内テキスト】
・メイン：親子清掃の確認リスト
・カード見出し：集合 / 持ち物 / 雨天時 / 来校方法
・結論帯：当日は車で来られません
```

### 画像生成プロンプトの方向性

縦長2:3。上部に日時と集合場所、中央に持ち物チェック、下部に雨天時と駐車場注意。保護者がスマホで保存して前日に見返せる構成にする。

## 例5：家庭文書 / 自由研究

### 入力文の例

```
朝顔の観察では、水の量を変えて成長の違いを比べました。
毎日100mlの鉢は葉が大きく育ち、毎日30mlの鉢は成長が遅くなりました。
水をあげすぎた鉢は土が乾かず、根元が弱くなりました。
結果として、多すぎても少なすぎてもよくなく、土の乾き具合を見ることが大切だと分かりました。
```

### 図解しおりの判定例

```
【選定パターン】
No.8 表形式比較型（大分類：比較型）

【推奨アスペクト比】
note本文・家庭保存向け 3:4 縦長

【図解密度】
作り込み（Showcase）

【リッチ化要素】
鉢植えモチーフ / 3条件比較 / 成長アイコン / 結論帯

【図解内テキスト】
・メイン：水の量で成長は変わる？
・カード見出し：30ml / 100ml / あげすぎ
・結論帯：土の乾き具合を見る
```

### 画像生成プロンプトの方向性

縦長3:4。3つの鉢を並べ、葉の大きさと土の状態で違いを見せる。下部に「多すぎても少なすぎてもNG」の結論帯。子どもが発表資料に貼っても分かりやすい明るいトーンにする。

## 例6：PM資料 / 要件定義の論点整理

### 入力文の例

```
要件定義で揉めるのは、機能一覧が足りないからだけではありません。
誰の業務を、どこまで、いつの時点で変えるのかが曖昧なまま進むと、後から「それは想定外でした」が増えます。
特に、現行業務、移行時の暫定運用、リリース後の定着支援を分けずに議論すると、判断すべき論点が混ざります。
最初に「対象業務」「移行範囲」「運用後の責任」を切り分けるだけで、会議の迷子がかなり減ります。
```

### 図解しおりの判定例

```
【選定パターン】
No.24 レイヤー型（大分類：構造型）

【推奨アスペクト比】
PM資料・note本文向け 3:4 縦長

【図解密度】
作り込み（Showcase）

【リッチ化要素】
業務フロー背景 / 3レイヤー構造 / 論点バッジ / 結論帯

【図解内テキスト】
・メイン：要件定義は層で分ける
・カード見出し：現行業務 / 移行範囲 / 運用責任
・結論帯：論点を混ぜると会議が迷子になる
```

### 画像生成プロンプトの方向性

縦長3:4。上部に「要件定義は層で分ける」のタイトル、中央に現行業務・移行範囲・運用責任の3レイヤーを積み、各層に代表論点を短いラベルで配置する。下部に「論点を混ぜると会議が迷子になる」の結論帯を置き、PM資料としてもnote本文としても読める密度にする。

## 直近note図解から反映した改善ポイント

- 画面キャプチャや商品モックが入ると、初心者にとって安心感が上がる
- キャラや人物がいると、比較や検証が「実験」として見える
- 結論帯がある図解は、読者が何を持ち帰ればいいか分かりやすい
- 横長3カラムはPCではきれいだが、noteスマホ本文では文字が小さくなりやすい
- 縦長2:3や3:4は、手順・比較・まとめを上から順に読ませやすい
- リッチ化は文字量を増やすことではなく、モチーフ・構図・視線誘導・結論帯で行う
