# 記事制作エージェントの役割分担表

チームビルディングの科学（および家庭ビルディングの科学）の記事制作における、統括エージェントと子エージェントの役割整理。
Claude Code用の定義は `prompt_vault/.claude/agents/`、Codex用の定義は `prompt_vault/.codex/agents/` に格納する。

---

## 設計方針：作成は統合、検証・画像QA・公開前チェックは分離

**作成系は統括エージェントが一貫して担当し、検証系だけを子エージェントに分離する。**
Codex環境では画像生成と生成画像チェックも分離し、作成者バイアスで「なんとなく良い画像」を採用しない。
公開直前の完成度チェック（pre-publish-checker）も、作成の文脈を持たない専任に分離し、「分かっているつもり」での取りこぼしを防ぐ。

### 作成を分散しない理由

- 子エージェントはゼロから起動するため、persona_guideline.md・CLAUDE.md・series_base_rules.md を人数分読み直すことになり、トークン消費が膨らむ
- 記事のコアメタファー（例：Vol.104の「予習係」）が本文・図解・SNS・プロンプトを貫いている必要があり、分業するとトーンが分裂するリスクがある
- 工程がほぼ直列（本文→図解→お土産プロンプト→テスト）で、並列化の時間短縮が小さい

### 検証を分離する理由

- 「書いた本人がチェックする」と、自分の出力が正しく見えるバイアスがかかる（Vol.104初稿のCTA漏れ・ヘッダートンマナ違反はこれが一因）
- チェック担当は作成の文脈を知らない初見の方が、読者目線の違和感を拾える
- 検証4種（記事監査・プロンプトQA・画像QA・公開前チェック）は互いに独立しており、タイミングが重なるものは並列実行できる

## 役割一覧

| エージェント | 種別 | 責務 | 入力 | 出力 | 禁止事項 |
|---|---|---|---|---|---|
| **統括**（メインのClaude Code/Codex） | メイン | テーマ受領、article.md・header_image.md・infographic_*.md・sns_post.md・prompt.md の作成、子エージェントの起動、指摘の修正、git push、完了報告 | ユーザーのテーマ・ネタ | vol[番号]/ 一式 | 自分で書いた記事・画像の最終チェックを自分だけで済ませない／prompt.md修正のユーザー確認を省略しない |
| **article-checker**（記事チェッカー） | 子（読み取り専用） | トンマナ監査。チェックリスト全項目の照合と重大度別指摘 | vol[番号]/ のパス | 監査レポート（チャット返答のみ） | ファイルの修正・作成 |
| **prompt-qa-tester**（プロンプトQAテスター） | 子 | qa_crash_tester の7カテゴリQA実施、tests/ 3ファイルの記録 | vol[番号]/prompt.md のパス | tests/test_cases.md・test_results.md・test_report.md＋判定報告 | prompt.md本体の上書き（修正差分案をレポートに書くまで） |
| **infographic-generator**（図解生成エージェント） | 子（Codex向け） | 図解化ポイントの抽出、画像生成プロンプト作成、画像生成、生成ログ整理 | 記事本文＋図解ルール＋キャラ参照 | 図解候補マップ・生成プロンプト・画像ファイル | 画像の採用確定／シリーズ外キャラ追加／本文にない断定の追加 |
| **zukaishiori-image-producer**（図解しおり画像制作） | 子（Codex向け） | 図解しおりPro版の判断ロジックで図解候補マップ作成・68型選定・比率/華やかさ/文字量の決定・画像生成プロンプト作成・生成ログ整理 | 記事本文＋図解しおりKnowledge＋キャラ参照 | image_assets.md・header_image.md・infographic_*.md・images/README.md（生成ログ） | 候補判定を飛ばしたプロンプト作成／へなP不使用（チーム版）／生成画像の採用確定 |
| **generated-image-checker**（生成画像チェッカー） | 子（Codex/Claude Code） | 内容一致・可読性・華やかさ・ブランド性・保存性で生成画像を判定。日本語文字崩れ・キャラ別人化の検出 | 生成画像＋元記事＋生成プロンプト | 採用判定レポート（チャット返答のみ） | 画像を見ずに採用判定／必要な再生成を軽微修正扱いにする |
| **pre-publish-checker**（公開前最終チェッカー） | 子（読み取り専用） | 公開直前のローカル最終チェック。プレースホルダー残り・URL未差し込み・画像の揃い・QA合格・リンク/表記整合 | vol[番号]/ のパス | 公開可否判定＋公開後にやることリスト（チャット返答のみ） | ファイルの修正・作成／トンマナ指摘（article-checkerと役割を分ける） |

※ 図解生成（infographic-generator）は、画像生成を自動実行できるCodex環境で優先運用する。Claude Code環境では画像生成が手動になるため、プロンプト作成までを統括が担い、ユーザーが生成する。
※ 生成画像の採用判定（generated-image-checker）は**Claude Codeでも実行できる**（画像ファイルをReadで実際に見て判定）。手動生成した画像も、記事に貼る前に必ず判定を通す。

## 制作フロー

```text
ユーザー（テーマ提供）
   │
   ▼
統括: 記事一式を作成
  article.md / header_image.md / infographic_*.md / sns_post.md / prompt.md
   │
   ├──▶ 子: article-checker（記事一式の監査）─── 監査レポート ──┐
   │                                                            │
   ├──▶ 子: prompt-qa-tester（prompt.mdのQA）── tests/ ＋判定 ──┤ ←2つは並列起動可
   │                                                            │
   ├──▶ 子: infographic-generator / zukaishiori-image-producer
   │      （Codexで図解設計〜画像生成）──────────── 画像 ──┐
   │                                                          │
   ├──▶ 子: generated-image-checker（生成画像QA）── 採用判定 ─┤ ←画像ありの場合
   │                                                          │
   ▼                                                            │
統括: 指摘・QA結果への対応 ◀─────────────────────────────────────┘
  - 記事の必須修正 → 修正して article-checker で再監査
  - QA不合格 → V2修正案を作成 → ★ユーザー確認★ → prompt.md に適用 → 再テスト
  - 画像QA不合格 → 図解プロンプト修正 → 再生成 → generated-image-checker で再確認
   │
   ▼
統括: git add → commit → push origin master
   │
   ▼
統括: 完了報告（監査結果・QA判定のサマリー付き）
   │
   ▼
ユーザー: 画像生成（手動。Codexでは infographic-generator が自動生成）
   │
   ├──▶ 子: generated-image-checker（画像をReadで判定）── 採用/再生成 ──┐
   │                                                                    │
統括: 再生成が必要なら図解プロンプト調整 → 再生成 → 再判定 ◀─────────────┘
   │
   ▼
公開フェーズ（詳細は team_building_science/CLAUDE.md「公開前・公開後チェック手順」）:
  ① 子: pre-publish-checker（ローカル最終チェック）→ 🔴ブロッカー解消
  ② 統括: note下書き確認（ユーザーが下書きプレビューを開き、Claude in Chromeで
     貼り付け崩れ・画像・有料ライン・スマホ表示を確認）
  ③ ユーザー: note公開
  ④ 統括: 公開直後確認（WebFetchでタグ認識・無料エリアの見え方・リンクを確認）
     → 公開URLを sns_post.md とメモリ（記事リンク一覧）へ反映
```

## 運用ルール

- article-checker は**記事一式が完成した時点**で必ず起動する（作成途中に部分監査させない）
- prompt-qa-tester は **prompt.md の作成・修正が完了した時点**で必ず起動する（test_report.md が「未実行」のまま公開しない）
- Codexで画像生成を行う場合は、infographic-generator または zukaishiori-image-producer の生成後に generated-image-checker を起動する
- 記事監査・プロンプトQA・画像QAは独立しているため、**並列起動してよい**
- 子エージェントへの依頼には「vol[番号]のフォルダパス」を明示する（子は会話の文脈を持たないため）
- 検証で指摘ゼロでも、統括は結果サマリーを完了報告に含める
- prompt.md の上書きは、QA結果に基づく場合でも**必ずユーザー確認を取ってから**行う
- 生成画像は記事に貼る前に generated-image-checker の採用判定を必ず通す（手動生成した画像も対象）
- pre-publish-checker は**noteに貼り付ける前（公開ボタンを押す前）**に起動し、🔴ブロッカーが残ったまま公開しない
- note下書きのWeb確認（②）と公開直後確認（③）は、ブラウザ操作とユーザーとの対話が必要なため子エージェントに切り出さず、統括が実行する
- Codexでは `.codex/agents/` の定義を読み、必要に応じて `multi_agent_v1.spawn_agent` に役割定義と対象パスを渡して子エージェントを起動する
