# 家庭ビルディングの科学 - 自動化ワークフロー

このファイルはClaude Codeが参照する自動化ルールです。
**必ず `persona_guideline.md` を先に読み込んでから作業を開始してください。**

本連載は『チームビルディングの科学』の兄弟マガジンです。構成のDNAは継承しますが、対象読者・トンマナは「家庭で頑張る主婦・ママ」向けに最適化されています。

---

## ワークフロー

### STEP 1: 記事下書き作成
1. `persona_guideline.md` を読み込む
2. ユーザーから提供されたテーマ・ネタを元に下書きを作成
3. `vol[番号]/article.md` に保存（フォルダがなければ作成）

### STEP 2: プロンプト作成（必要な場合）
1. `vol[番号]/prompt.md` を作成
2. 「明日から使えるお土産」として記事から相対パス（`./prompt.md`）で参照できるようにする

### STEP 3: Threads投稿文作成
1. `vol[番号]/sns_post.md` を作成
2. 投稿1（本文＋画像指定）と投稿2（URL返信）の連投形式で記載
3. トピックを1つ選定して記載する（ハッシュタグは使用しない）

### STEP 4: 完了報告
- 下書きの保存場所
- プロンプトの保存場所
- Threads投稿文の保存場所
- 次のアクション（必要があれば）

## ファイル命名規則

Volごとに専用フォルダを作り、必要な成果物をすべてその中にまとめる（チームビルディングの科学と統一）。
ファイル名はシンプル名（`article.md`／`prompt.md` など）を使うこと。

```
vol[番号]/                  ← 1Volごとにフォルダをまとめてここへ
  article.md                ← 記事本文
  prompt.md                 ← メインのお土産プロンプト（1本の場合）
  header_image.md           ← ヘッダー画像生成プロンプト
  infographic.md            ← 図解生成プロンプト（1本にまとめる場合）
  infographic_[名前].md     ← 図解生成プロンプト（複数に分ける場合）
  manga.md                  ← 漫画スクリプト・生成プロンプト（ある場合）
  sns_post.md               ← Threads投稿文
  prompt_[名前].md          ← メインプロンプトと別に補助プロンプトがある場合

_shared/                    ← Volに依存しない共通プロンプト
  magazine_header_prompt.md ← マガジン全体ヘッダー画像
```

番号は3桁ゼロパディング（vol001, vol002, ...）。
旧構造（`articles/`, `prompts/`, `sns/` 配下にVol番号付きファイル名で配置）は使わない。

## 漫画（4コマ）の構成ルール

- **問題と解決を必ずセットで描く**：冒頭漫画は「共感（問題シーン）」だけで終わらせない。AIや提案手法を使って状況が好転するシーンまで含め、読者がSNSで漫画単体を見ても価値が伝わる完結した物語にすること。
- **ページ数は1ページに限定しない**：4コマ1ページに収まらない場合は2〜3ページに分けてよい。Page1＝問題・共感、Page2＝解決・達成感、という構成が標準。
- **解決シーンに必ず「プロンプト・AI起動」の描写を含める**：解決ページでは、実際にプロンプトを起動して子どもに渡し、親がタスクを完了するシーンを入れること。抽象的な「解決した」イメージではなく、具体的な操作→結果の流れを描く。
- **複数ページは別画像として生成**：2ページ構成の場合、ページ1とページ2はそれぞれ別の画像プロンプトとして記述し、別々に生成すること。
- **家事全般・感情負荷テーマでは、冒頭漫画を推奨**：献立や子育てのような具体タスクよりも「なんか全部大変」「しんどさの正体が分からない」という抽象テーマでは、漫画で生活シーンを先に可視化してから本文に入ると読者の理解が早い。
- **漫画内の解決は“家事が消える”ではなく“順番が見える”にする**：AI活用後も家事そのものは残る。誇張を避け、AIによって「今日やること・明日に送ること・頼むこと」が整理され、表情と空気が軽くなる描写にすること。

## 重要な注意事項

- **persona_guideline.md は毎回必ず読む**（煽り表現・呪いの再生産チェックのため）
- **「家事は愛情」「お母さんの手作りが一番」という昭和の呪いを絶対に再生産しない**
- **問題は読者の人格ではなく科学的メカニズムに帰属させる**
- **プロンプトの上書きは必ずユーザー確認を取ってから行う**
- **チームビルディングの科学のビジネス用語をそのまま流用しない**（家庭シーンに翻訳すること）
- **プロンプトを更新する際はファイル名を変えずに上書きすること**（例：`_v2` などのサフィックスを付けない）。新旧の差分はgitの履歴で管理する。バージョン違いのファイルが並存すると比較が困難になるため厳禁。

## Threads投稿ルール

### フォーマット
- **連投形式:** 投稿1（本文）→ 投稿2（URL返信）の2投稿セット
- **URL:** 本文には入れない。返信形式の投稿2にのみ記載
- **ハッシュタグ:** 使用しない（Threadsでは効果が薄い）
- **トピック:** 1つだけ選定して記載する（例：料理、家事、AI）
- **画像:** 記事に対応する漫画または図解を1枚目に添付

### トンマナ（必須）
- **スタイル:** 読者への共感型（Bスタイル）。告白・失敗談スタイルは使わない
  - 理由：著者は料理・育児の当事者ではないため、告白スタイルは不自然
  - 「料理もしない私が言うのも何ですが」のような自己開示・言い訳も不要
- **語調:** カジュアル・日常会話調。文学的・詩的な表現は使わない
- **文体の基準:** 断定的・直接的な話し言葉。「〜らしいです」「〜って言うんですね」より「〜です」「〜って言います」が自然
- **冒頭:** `おはへな～☀️` の挨拶から始める（独自の定型挨拶）。「今回は〜を書きました」は禁止
- **絵文字:** 感情・状況に合わせて自然に使用。1投稿に複数使ってよい（過剰は避ける）。文末の矢印は `👇`
- **締め:** `〜ますように～🙏` という願い・祈りの形で締める（例：「負荷が下がりますように～🙏」）
- **長さ:** 投稿1は読み切れる長さ（スクロールしすぎない）

### トピック選定の目安
| 記事テーマ | 推奨トピック |
|-----------|------------|
| 献立・料理 | 料理 |
| 家事全般・時短 | 家事 |
| 子育て | 子育て |
| AI活用全般 | AI |

## トンマナ チェックリスト（記事提出前）

記事を作成したら、以下を必ず確認してください。

- [ ] 冒頭は家庭シーン描写または心の声で始まっているか
- [ ] 「家庭ビルディングの科学・Vol.XXX となる今回は〜」という定型導入句があるか
- [ ] 科学的メカニズムの見出しがあり、認知バイアス名が**太字＋日本語名＋英語名**で記載されているか
- [ ] 「愛ある厳しさ」セクションで読者の反論を先取りして切り返しているか
- [ ] 「【今回の明日から使えるお土産】」という定型句があるか
- [ ] プロンプトへの相対リンク `[→ プロンプトを見る](./prompt.md)` があるか
- [ ] 締めのセクションがメタファー形式の見出しか
- [ ] 「あなた」への呼びかけで読者への共感・免責表現があるか
- [ ] 絵文字は記事本文に使用していないか（プロンプト内のSystem Messageは除く）
- [ ] 情報商材的な誇張表現（「絶対」「魔法のように」等）が入っていないか
- [ ] 禁止語リスト（「主婦の味方」「手抜き」「家事は愛情」等）を使っていないか
- [ ] 「あなたのせいじゃない」というメッセージが伝わるか
- [ ] タグ行の直前にスキ＆フォローCTAがあるか（「スキ（♡）」と「フォロー」への呼びかけ）
- [ ] Tags行が末尾にあり、`#タグ名 #タグ名` のハッシュタグ形式になっているか（カンマ区切り不可・noteではタグとして機能しない）
