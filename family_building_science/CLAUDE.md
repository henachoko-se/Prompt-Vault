# 家庭ビルディングの科学 - 自動化ワークフロー

このファイルはClaude Codeが参照する自動化ルールです。
**必ず `persona_guideline.md` を先に読み込んでから作業を開始してください。**
記事・図解・画像生成・SNS展開では、リポジトリ直下の `series_base_rules.md` も合わせて参照してください。

本連載は『チームビルディングの科学』の兄弟マガジンです。構成のDNAは継承しますが、対象読者・トンマナは「家庭で頑張る主婦・ママ」向けに最適化されています。

---

## ワークフロー

### STEP 1: 記事下書き作成
1. `persona_guideline.md` を読み込む
2. ユーザーから提供されたテーマ・ネタを元に下書きを作成
3. `vol[番号]/article.md` に保存（フォルダがなければ作成）

### STEP 2: プロンプト作成（必要な場合）
1. `vol[番号]/prompt.md` を作成
2. 「明日から使えるお土産」として記事から相対パス（`./prompt.md`）で参照できるようにする

### STEP 3: プロンプトのQAテスト（必須）
1. `../team_building_science/_tools/qa_crash_tester.md` のロジックに従い、`vol[番号]/prompt.md` に対するテストケースを設計する
2. 家庭ビルディングの科学のプロンプトは原則**対話型・フェーズ管理型**として扱う（負荷テストは「混在入力テスト」形式）
3. Claude Code自身がプロンプトのロジックを内部実行し、各テストケースの出力を検証する
4. 結果を評価し、合格なら `vol[番号]/qa_report.md` に合格レポートを保存して次STEPへ進む
5. 不合格（Critical/Major）の場合は改善版プロンプトを生成してユーザーに確認を求め、承認後に `prompt.md` を上書き更新する（`_v2` サフィックス禁止）
6. **プロンプトがないVolは本STEPをスキップしてよい**

### STEP 4: Threads投稿文作成
1. `vol[番号]/sns_post.md` を作成
2. 投稿1（本文＋画像指定）と投稿2（URL返信）の連投形式で記載
3. トピックを1つ選定して記載する（ハッシュタグは使用しない）

### STEP 5: noteヘッダー・記事内図解プロンプト作成（必須）
1. 新規Vol記事を作成した場合は、ユーザーから個別に指示がなくても `vol[番号]/header_image.md` を必ず作成する
2. 新規Vol記事を作成した場合は、ユーザーから個別に指示がなくても `vol[番号]/infographic.md` を必ず作成する
3. `header_image.md` には、noteヘッダー画像用の生成プロンプトを記載する
   - noteヘッダー画像のアスペクト比は必ず **1.91:1** にする
   - 推奨サイズは **1280×670px** または **1910×1000px**。生成AIの出力がずれた場合は、保存前にトリミングまたは余白調整で **1.91:1** に補正する
   - 保存後に実ファイルのピクセルサイズを確認し、`幅 ÷ 高さ = 1.91前後` になっていることを完了条件にする
   - 対応記事への相対リンクを入れる
   - へなちょこパパのキャラクター参照画像パスを入れる
4. `infographic.md` には、記事内に挿入する図解プロンプトをまとめる
   - 原則として、漫画または図解を含む **3〜5種** を用意する
   - 記事内の挿入位置、目的、形式を明記する
   - 生成後のチェックリストを入れる
5. 記事本文に図解挿入プレースホルダーがある場合は、`infographic.md` の図解番号・タイトルと対応させる

### STEP 5.5: 記事用画像生成ルール（絶対遵守）
1. 記事用画像を作る前に、必ず該当Volの以下を読み直す
   - `article.md`
   - `header_image.md`
   - `infographic.md`
   - 漫画がある場合は `manga.md` または `infographic.md` 内の漫画台本・YAML
2. `header_image.md` はnoteヘッダー画像生成の正本である。ヘッダー画像はこのプロンプトを元に画像生成AIで作る
3. `infographic.md` は記事内図解・漫画画像生成の正本である。図解画像はこのファイルの各プロンプトを元に画像生成AIで作る
4. 漫画は必ず漫画台本およびStep2用YAML/出力プロンプトを利用して生成する。記事本文から直接漫画画像を生成しない
5. `infographic.md` に画像生成プロンプトが用意されている図解を、独自のSVG・HTML・PowerPoint風図解・簡易チャート・コード生成画像で代替してはいけない
6. 「日本語テキストの正確性が不安」などの理由で、勝手にダサい簡易図へ逃げない。必要なら画像生成プロンプト側を改善し、再生成する
7. 画像生成後に文字崩れ・構図崩れ・品質不足がある場合は、生成物を正式版扱いにせず、以下のどちらかにする
   - `infographic.md` の該当プロンプトを改善して再生成する
   - ユーザーに「この画像は文字崩れがあるため再生成が必要」と報告する
8. 記事本文の図解プレースホルダーを画像リンクへ差し替えるのは、ユーザーから明示的に依頼された場合、または画像の品質確認が完了して正式採用が決まった場合のみとする
9. 生成画像を保存する場合は、`vol[番号]/images/` に用途が分かる名前で保存する
   - さらに、記事で利用する正式画像は共有・投稿用の画像置き場 `C:\Users\henac\OneDrive\ドキュメント\AI\へなちょこPM\へなちょこパパ` にもコピーして保存する
   - 共有・投稿用の保存名は、既存Volと同じく `Vol004note.png`、`Vol004漫画P1.png`、`Vol004【図解2】...png` のように、Vol番号と用途が分かる日本語名にする
   - **共有・投稿用フォルダがnote投稿時の正本である。** リポジトリ内の `assets/` に保存しただけで完了扱いにしない
   - 図解や漫画を作り直した場合は、共有・投稿用フォルダ側も同じタイミングで同じ用途名のファイルへ上書きする。英字のリポジトリ用ファイル名だけで置かない
   - 共有・投稿用フォルダに古い同Vol画像やラフ画像が残り、正式画像と紛らわしい場合は、削除せず `_VolXXX_旧画像_YYYYMMDD_非使用` などの退避フォルダへ移す
   - 作業完了前に必ず共有・投稿用フォルダで `VolXXX*` の一覧を確認し、noteに利用する正式画像だけが分かる状態になっていることを報告する
   - ラフ・失敗・下書き画像は共有・投稿用フォルダへコピーしない
   - 例：`header.png`, `manga_page1.png`, `infographic_02.png`
10. ラフ・失敗・代替案として生成した画像は、正式画像と混ざらないように `images/_drafts/` などへ分ける。正式画像として記事・SNS・PromptVaultに出さない

### STEP 6: 漫画台本・画像生成フロー（必須）
1. 記事内図解に漫画を含める場合は、ユーザーから個別に指示がなくても `vol[番号]/manga.md` を作成する
2. 漫画台本は `team_building_science/_tools/step1_script_maker_kai.md` のルールに従う
   - 台本（ネーム）を先に作る
   - ページ数、綴じ、モード、目的を明記する
   - 登場人物の外見を全ページで表記ゆれなく統一する
   - コマごとに技法、視覚的指示、優先度を入れる
3. 漫画画像生成用YAMLは `team_building_science/_tools/step2_image_gen_kai.md` に渡せる形式で記載する
4. 画像を生成する場合も、Step1で作ったYAMLをStep2画像生成エンジンへ渡す前提で行う。記事内容から直接漫画画像を生成しない
5. 漫画画像にはページ番号、コマ番号、Panel番号、Page表記、コマ見出し、YAML項目名、制作メモを描かせない。これらは制作管理用であり、読者向け画像に表示しない
6. 直接生成した漫画画像がある場合は、正式版ではなくラフ生成物として明記し、正式な漫画画像はStep1/Step2フローで再生成する

### STEP 7: PromptVault反映（必須）
1. 新規Volフォルダを作成した場合は、ローカル保存だけで完了扱いにしない
2. `git status --short` で新規ファイル・変更ファイルを確認する
3. 原則として当該Volフォルダだけを `git add` する（無関係な既存変更は触らない）
4. `git commit` で保存し、`git push origin master` でGitHubへ反映する
5. クラウド版PromptVaultで表示させる必要がある場合は、Cloud Runへ再デプロイする
6. デプロイは、作業ツリーに無関係な未コミット変更がある場合、最新GitHub状態を一時クローンしてそこから実行する
7. 反映後、PromptVaultの画面を再読み込みすれば `volXXX` が出る状態まで確認する

補足：
- ローカルの `C:\Users\henac\prompt_vault` にファイルを作っただけでは、クラウド版PromptVaultには出ない
- Cloud Run本番はコンテナ内のファイルを見ているため、GitHub pushだけでは即時表示されない場合がある
- `gcloud` が証明書エラーで止まる場合は、原因を明記し、必要に応じて一時対応後に必ず設定を戻す

### STEP 8: 完了報告
- 下書きの保存場所
- プロンプトの保存場所
- QAテスト結果（合格 or 改善点サマリー）
- noteヘッダー画像プロンプトの保存場所
- 記事内図解プロンプトの保存場所
- 漫画台本・Step2用YAMLの保存場所
- 画像生成時に `header_image.md` / `infographic.md` / 漫画台本・YAMLを正本として使ったか
- 独自SVG・簡易図・コード生成画像で代替していないか
- Threads投稿文の保存場所
- GitHubへpushしたか
- Cloud Runへデプロイしたか

## ファイル命名規則

Volごとに専用フォルダを作り、必要な成果物をすべてその中にまとめる（チームビルディングの科学と統一）。
ファイル名はシンプル名（`article.md`／`prompt.md` など）を使うこと。

```
vol[番号]/                  ← 1Volごとにフォルダをまとめてここへ
  article.md                ← 記事本文
  prompt.md                 ← メインのお土産プロンプト（1本の場合）
  qa_report.md              ← QAテストレポート（qa_crash_tester結果）
  header_image.md           ← ヘッダー画像生成プロンプト
  infographic.md            ← 図解生成プロンプト（1本にまとめる場合）
  infographic_[名前].md     ← 図解生成プロンプト（複数に分ける場合）
  manga.md                  ← 漫画スクリプト・生成プロンプト（ある場合）
  sns_post.md               ← Threads投稿文
  prompt_[名前].md          ← メインプロンプトと別に補助プロンプトがある場合

_shared/                    ← Volに依存しない共通プロンプト
  magazine_header_prompt.md ← マガジン全体ヘッダー画像
```

番号は3桁ゼロパディング（vol001, vol002, ...）。
旧構造（`articles/`, `prompts/`, `sns/` 配下にVol番号付きファイル名で配置）は使わない。

## 漫画（4コマ）の構成ルール

- **問題と解決を必ずセットで描く**：冒頭漫画は「共感（問題シーン）」だけで終わらせない。AIや提案手法を使って状況が好転するシーンまで含め、読者がSNSで漫画単体を見ても価値が伝わる完結した物語にすること。
- **ページ数は1ページに限定しない**：4コマ1ページに収まらない場合は2〜3ページに分けてよい。Page1＝問題・共感、Page2＝解決・達成感、という構成が標準。
- **解決シーンに必ず「プロンプト・AI起動」の描写を含める**：解決ページでは、実際にプロンプトを起動して子どもに渡し、親がタスクを完了するシーンを入れること。抽象的な「解決した」イメージではなく、具体的な操作→結果の流れを描く。
- **複数ページは別画像として生成**：2ページ構成の場合、ページ1とページ2はそれぞれ別の画像プロンプトとして記述し、別々に生成すること。
- **家事全般・感情負荷テーマでは、冒頭漫画を推奨**：献立や子育てのような具体タスクよりも「なんか全部大変」「しんどさの正体が分からない」という抽象テーマでは、漫画で生活シーンを先に可視化してから本文に入ると読者の理解が早い。
- **漫画内の解決は“家事が消える”ではなく“順番が見える”にする**：AI活用後も家事そのものは残る。誇張を避け、AIによって「今日やること・明日に送ること・頼むこと」が整理され、表情と空気が軽くなる描写にすること。

## 重要な注意事項

- **persona_guideline.md は毎回必ず読む**（煽り表現・呪いの再生産チェックのため）
- **「家事は愛情」「お母さんの手作りが一番」という昭和の呪いを絶対に再生産しない**
- **問題は読者の人格ではなく科学的メカニズムに帰属させる**
- **プロンプトの上書きは必ずユーザー確認を取ってから行う**
- **チームビルディングの科学のビジネス用語をそのまま流用しない**（家庭シーンに翻訳すること）
- **プロンプトを更新する際はファイル名を変えずに上書きすること**（例：`_v2` などのサフィックスを付けない）。新旧の差分はgitの履歴で管理する。バージョン違いのファイルが並存すると比較が困難になるため厳禁。

## Threads投稿ルール

### フォーマット
- **連投形式:** 投稿1（本文）→ 投稿2（URL返信）の2投稿セット
- **URL:** 本文には入れない。返信形式の投稿2にのみ記載
- **ハッシュタグ:** 使用しない（Threadsでは効果が薄い）
- **トピック:** 1つだけ選定して記載する（例：料理、家事、AI）
- **画像:** 記事に対応する漫画または図解を1枚目に添付

### トンマナ（必須）
- **スタイル:** 読者への共感型（Bスタイル）。告白・失敗談スタイルは使わない
  - 理由：著者は料理・育児の当事者ではないため、告白スタイルは不自然
  - 「料理もしない私が言うのも何ですが」のような自己開示・言い訳も不要
- **語調:** カジュアル・日常会話調。文学的・詩的な表現は使わない
- **文体の基準:** 断定的・直接的な話し言葉。「〜らしいです」「〜って言うんですね」より「〜です」「〜って言います」が自然
- **冒頭:** `おはへな～☀️` の挨拶から始める（独自の定型挨拶）。「今回は〜を書きました」は禁止
- **絵文字:** 感情・状況に合わせて自然に使用。1投稿に複数使ってよい（過剰は避ける）。文末の矢印は `👇`
- **締め:** `〜ますように～🙏` という願い・祈りの形で締める（例：「負荷が下がりますように～🙏」）
- **長さ:** 投稿1は読み切れる長さ（スクロールしすぎない）

### トピック選定の目安
| 記事テーマ | 推奨トピック |
|-----------|------------|
| 献立・料理 | 料理 |
| 家事全般・時短 | 家事 |
| 子育て | 子育て |
| AI活用全般 | AI |

## トンマナ チェックリスト（記事提出前）

記事を作成したら、以下を必ず確認してください。

- [ ] 冒頭は家庭シーン描写または心の声で始まっているか
- [ ] 「家庭ビルディングの科学・Vol.XXX となる今回は〜」という定型導入句があるか
- [ ] 科学的メカニズムの見出しがあり、認知バイアス名が**太字＋日本語名＋英語名**で記載されているか
- [ ] 「愛ある厳しさ」セクションで読者の反論を先取りして切り返しているか
- [ ] 「【今回の明日から使えるお土産】」という定型句があるか
- [ ] プロンプトへの相対リンク `[→ プロンプトを見る](./prompt.md)` があるか
- [ ] 締めのセクションがメタファー形式の見出しか
- [ ] 「あなた」への呼びかけで読者への共感・免責表現があるか
- [ ] 絵文字は記事本文に使用していないか（プロンプト内のSystem Messageは除く）
- [ ] 情報商材的な誇張表現（「絶対」「魔法のように」等）が入っていないか
- [ ] 禁止語リスト（「主婦の味方」「手抜き」「家事は愛情」等）を使っていないか
- [ ] 「あなたのせいじゃない」というメッセージが伝わるか
- [ ] タグ行の直前にスキ＆フォローCTAがあるか（「スキ（♡）」と「フォロー」への呼びかけ）
- [ ] Tags行が末尾にあり、`#タグ名 #タグ名` のハッシュタグ形式になっているか（カンマ区切り不可・noteではタグとして機能しない）
